### Solution

Just another multi-stage cipher from the *Lost in translation* series.

The first thing that came to my mind was to convert those numbers to ASCII characters. Handling numbers as decimal based didn't work well, but octal base did the job.

> MJRWKZTHGFTTEIDEMVTCAYTDNIQGCYTDMRSWMZZRM4ZCAZZRM4ZCAYLKNQQGCYTDMRSWM3JAMFUWYIDCMNSWMZZRM4ZCAZZRM4ZCAYLCMNSGKIDBMRVGWIDCMNVCAZLGM4YWU3JAM4YWOMRAMFRGGZDFEBSWMZZRNJWSAYLDMRTTE2BAMFRGGZDJNQQGOMLHGIQGCY3EMVTGOMRAMFRGKZTHGFTTEIDBMRSWMZZRM4ZCAYLCMNSGOMTJNQQGOMLHGIQGCY3EMVTGOMRAMFRGGZDFEBQWEZLGM4YWOMRAMJRWIZLGEA======

Based on the alphabet and characteristic padding, it was clear that the Base32 encoding was used in the second phase.

> bcefg1g2 def bcj abcdefg1g2 g1g2 ajl abcdefm ail bcefg1g2 g1g2 abcde adjk bcj efg1jm g1g2 abcde efg1jm acdg2h abcdil g1g2 acdefg2 abefg1g2 adefg1g2 abcdg2il g1g2 acdefg2 abcde abefg1g2 bcdef

By searching for some of the character sequences on Google, I found that the last stage used the 14-segment display encoding. [This decoder](http://kryptografie.de/kryptografie/chiffre/14-segment.htm) worked flawlessly.

```
HL18-7QTH-JZ1K-JKSD-GPEB-GJPU
```
